###  What algorithms are used to recommend the funds?

● The system uses two complementary algorithms that run in parallel, both sharing the same scored fund universe.

  ---
  The Recommendation System — Two Algorithms

  Algorithm 1 is a rule-based weighted scoring model with fixed template-driven allocations — simple, explainable, and SEBI-defensible.
  Algorithm 2 is a Sharpe-Optimised method that derives asset class weights dynamically from each fund's risk-adjusted return.
  Both use the same scored instrument universe. Their outputs are pooled, deduplicated by distinctness, ranked by Portfolio Sharpe
  Approx, and the best 4 options are presented. No machine learning, no covariance matrices — all computations are deterministic
  and auditable.

  ---
  Algorithm 1 — Weighted Scoring + Template-Driven Allocation

  Stage 1 — Score every instrument (0–100)

  Each fund gets a composite score from three components:

  Return Score — weight 40%
  min(cagr_3yr_pct / 20.0, 1.0) × 100
  A fund with 20% 3-year CAGR scores 100. One with 10% scores 50. Capped at 100 so outliers don't dominate.

  Risk Efficiency Score — weight 30%
  min(cagr_3yr_pct / max(volatility_pct, 0.5), 3.0) / 3.0 × 100
  This is a Sharpe-ratio-like measure — return per unit of volatility. The ratio is capped at 3.0 (normalised to 100). A fund returning 18%
  with 6% volatility scores higher than one returning 20% with 15% volatility. For static instruments (PPF, FDs, NSC) where volatility is zero,
   a fixed score of 75 is assigned.

  Capital Protection Score — weight 30%
  max(0, min(100, 100 − |max_drawdown_pct| × 2))
  Penalises for historical peak-to-trough losses. A fund with 0% drawdown scores 100. A −25% drawdown scores 50. A −50% drawdown scores 0.
  Static instruments (no market risk) always score 100 here.

  Final composite:
  Score = (Return × 0.40) + (RiskEfficiency × 0.30) + (CapProtection × 0.30)

  ---
  Stage 2 — Filter the universe

  Before ranking:
  - Instruments the client already holds in the same category are removed (get_portfolio_holdings)
  - EPF is excluded at the SQL layer entirely (not investable independently)
  - Instruments with no return data and not of STATIC type are marked data_unavailable and excluded
  - SAFE-class instruments (PPF, NSC, FDs, RBI Bonds) are always present in the universe regardless of the client's
    risk tier. The fund universe query applies `(risk_tier IN (...eligible tiers...) OR asset_class = 'SAFE')` so
    that guaranteed-return instruments are available to any client — including MEDIUM and HIGH risk tiers that would
    otherwise not include NO_RISK-tagged funds.

  ---
  Stage 3 — Pick a strategy template

  Based on the client's risk_preference_tier, a named strategy is selected with pre-defined asset class weights. For example:

  ┌───────────┬──────────────────┬──────────────────────────────────────────────────────────┐
  │ Risk Tier │     Strategy     │                        Allocation                        │
  ├───────────┼──────────────────┼──────────────────────────────────────────────────────────┤
  │ HIGH      │ Growth Maximiser │ 65% Equity / 15% International / 10% Gold / 10% Safe     │
  ├───────────┼──────────────────┼──────────────────────────────────────────────────────────┤
  │ MEDIUM    │ Growth Tilt      │ 50% Equity / 15% Hybrid / 15% Gold / 10% Safe / 10% Debt │
  ├───────────┼──────────────────┼──────────────────────────────────────────────────────────┤
  │ LOW       │ Income Focus     │ 45% Safe / 45% Debt / 10% Gold                           │
  ├───────────┼──────────────────┼──────────────────────────────────────────────────────────┤
  │ NO_RISK   │ FD Ladder        │ 90% Safe (FDs) / 10% Gold                                │
  └───────────┴──────────────────┴──────────────────────────────────────────────────────────┘

  Up to 4 strategies are defined per tier, each with different weights. The system generates only strategies that produce genuinely distinct
  portfolios — different fund picks or >10% difference in any bucket.

  ---
  Stage 4 — Fill each bucket

  For each asset class bucket in the strategy, the top-scoring instrument from Stage 1 is picked for that slot. If a single bucket would exceed
   35% of the portfolio, it's split across two instruments in the same class to avoid concentration.

  The allocation percentage per instrument is fixed by the strategy template, not computed dynamically. Then:

  suggested_amount = investable_amount × (allocation_pct / 100)
  Rounded to nearest Rs. 1,000, with the largest bucket absorbing any rounding remainder so the total always sums to the investable amount.

  ---
  Algorithm 2 — Sharpe-Optimised Allocation (Simplified MVT)

  Unlike Algorithm 1's pre-defined templates, Algorithm 2 derives asset class weights dynamically from each fund's Sharpe proxy.
  It produces exactly ONE portfolio option named "Sharpe Maximiser" (algorithm = "SHARPE_OPTIMISED").

  Stage 1 — Compute per-fund Sharpe proxy
    For market-traded funds (volatility_pct > 0):
      sharpe_i = cagr_3yr_pct / max(volatility_pct, 0.5)
    For static instruments (PPF, FDs, NSC — volatility = 0):
      sharpe_i = cagr_3yr_pct / 1.5
      (1.5 is a synthetic volatility floor to avoid infinite Sharpe for guaranteed-return instruments)

  Stage 2 — Derive class representative and raw weights
    For each asset class, the fund with the highest composite score (from Algorithm 1) is the class representative.
    Its sharpe_i becomes the class Sharpe (sharpe_class).
      sum_sharpe = Σ sharpe_class  (across all represented asset classes)
      raw_weight_class = (sharpe_class / sum_sharpe) × 100

  Stage 3 — Apply risk-tier floor/ceiling constraints
    Raw weights are hard-constrained to the client's declared risk tier.
    The client's risk preference is read-only — never overridden.

    ┌──────────┬──────────────────────────────────────────────────────────┐
    │ Risk Tier│ Constraints Applied                                      │
    ├──────────┼──────────────────────────────────────────────────────────┤
    │ NO_RISK  │ EQUITY ceiling = 0%, HYBRID ceiling = 0%, SAFE floor=55% │
    ├──────────┼──────────────────────────────────────────────────────────┤
    │ LOW      │ EQUITY ceiling = 15%, SAFE+DEBT combined floor = 60%     │
    ├──────────┼──────────────────────────────────────────────────────────┤
    │ MEDIUM   │ EQUITY floor = 10%, EQUITY ceiling = 60%                 │
    ├──────────┼──────────────────────────────────────────────────────────┤
    │ HIGH     │ EQUITY floor = 25%, EQUITY ceiling = 80%                 │
    └──────────┴──────────────────────────────────────────────────────────┘
    Surplus from ceiling violations is redistributed to GOLD or DEBT.
    All weights re-normalised to sum exactly 100% after constraints are applied.

  Stage 4 — Round and fill buckets
    Weights are rounded to the nearest 5%.
    Rounding remainder is added/subtracted from the largest non-zero bucket to maintain exactly 100%.
    Top-scoring fund per asset class (from Algorithm 1) fills each bucket.
    If a bucket exceeds 35% it is split across the top two funds in that class to avoid concentration.

  ---
  Option-Level Return Projections (both algorithms)

  For every option from either algorithm, five aggregate metrics are computed:

    Expected Portfolio Return    = round( Σ (alloc_pct / 100 × cagr_3yr_pct) , 2 )          [unit: % p.a.]
    Weighted Max Drawdown        = round( −1 × Σ (alloc_pct / 100 × |max_drawdown_pct|) , 2 )
    Portfolio Sharpe Approx      = expected_return / max(|weighted_drawdown|, 0.5)
    Projected Corpus (3yr)       = investable_amount × (1 + expected_return / 100)³           [nearest Rs. 1,000]
    Projected Net Gain           = projected_corpus − investable_amount                        [nearest Rs. 1,000]

  All corpus and gain amounts are formatted in Indian notation (e.g. "Rs.34,52,000"; ≥ 1 crore: "Rs.1,23,45,000").

  Implementation note: All five metrics are recomputed deterministically by a Python post-processor
  (_fix_wr_metrics in main.py) after the LLM returns the raw option structure. The LLM determines fund selection
  and option structure; Python corrects the numbers. This ensures arithmetic accuracy and consistent corpus values
  regardless of LLM calculation fidelity. Sorting by Portfolio Sharpe Approx descending and renumbering options
  1–4 is also performed in this post-processing pass.

  ---
  How Both Algorithms Are Combined

  After both algorithms produce their options:
  1. Pooling: all options are combined (up to 3 Template-Driven options + 1 Sharpe-Optimised option).
  2. Distinctness check: two options are not distinct if they share all the same fund IDs AND every asset class allocation
     bucket differs by less than 10 percentage points. The option with the lower Portfolio Sharpe Approx is dropped.
     On a tie, the Sharpe-Optimised option is preferred.
  3. Ranking: all surviving options sorted by Portfolio Sharpe Approx descending.
  4. Trim: at most 4 options retained and renumbered 1–4.
  The Sharpe-Optimised option competes on equal footing — it wins a slot only if its risk-adjusted return ranks in the top 4.

  ---
  What it is NOT

  - Not Markowitz / Modern Portfolio Theory — no covariance matrix, no efficient frontier, no mean-variance optimisation
  - Not Black-Litterman — no market equilibrium returns, no investor views model
  - Not ML — no model trained on historical data, no clustering, no collaborative filtering
  - Not factor-based — no momentum, value, or quality factor tilts

  ---
  Why this approach

  Three reasons:

  1. Explainability — Every recommendation carries a rationale citing the exact 3-year CAGR and max drawdown. A compliance auditor or the
  client themselves can verify why a fund was picked. An ML model can't offer that.
  2. Regulatory defensibility — SEBI requires suitability documentation. A weighted formula with disclosed parameters is auditable. A neural
  network is not.
  3. Deterministic enough to be trustworthy — The same fund universe + same client + same risk tier should produce consistent recommendations.
  A scoring formula delivers that; an LLM left to freely pick funds would not.
